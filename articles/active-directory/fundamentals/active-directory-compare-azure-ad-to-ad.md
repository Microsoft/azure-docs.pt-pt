# <a name="compare-active-directory-to-azure-active-directory"></a>Compare o Diretório Ativo com o Diretório Ativo azure

O Azure Ative Directory é a próxima evolução das soluções de gestão de identidade e acesso para a nuvem. A Microsoft introduziu os Ative Directory Domain Services no Windows 2000 para dar às organizações a capacidade de gerir múltiplos componentes e sistemas de infraestrutura no local utilizando uma única identidade por utilizador.

A Azure AD aborda esta abordagem para o próximo nível, fornecendo às organizações uma solução Identidade como Serviço (IDaaS) para todas as suas aplicações através da nuvem e no local.

A maioria dos administradores de TI estão familiarizados com os conceitos de Serviços de Domínio de Diretório Ativo. A tabela que se segue descreve as diferenças e semelhanças entre conceitos de Diretório Ativo e Diretório Ativo Azure.

|Conceito|Diretório Ativo (AD)|Azure Active Directory |
|:-|:-|:-|
|**Utilizadores**|||
|Provisionamento: utilizadores | As organizações criam utilizadores internos manualmente ou utilizam um sistema de provisionamento interno ou automatizado, como o Microsoft Identity Manager, para se integrarem com um sistema de RH.|As organizações ad existentes usam [o Azure AD Connect](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-sync-whatis) para sincronizar identidades na nuvem.</br> A Azure AD adiciona suporte para criar automaticamente utilizadores a partir de [sistemas de RH em nuvem](https://docs.microsoft.com/azure/active-directory/saas-apps/workday-tutorial). </br>O Azure AD pode fornecer identidades em aplicações [SCIM habilitadas](https://docs.microsoft.com/azure/active-directory/manage-apps/use-scim-to-provision-users-and-groups) a fornecer automaticamente às aplicações os detalhes necessários para permitir o acesso aos utilizadores. |
|Provisionamento: identidades externas| As organizações criam utilizadores externos manualmente como utilizadores regulares numa floresta de AD externa dedicada, resultando em despesas gerais da administração para gerir o ciclo de vida das identidades externas (utilizadores convidados)| A Azure AD fornece uma classe especial de identidade para apoiar identidades externas. [O Azure AD B2B](https://docs.microsoft.com/azure/active-directory/b2b/) irá gerir a ligação à identidade externa do utilizador para se certificar de que são válidos. |
| Gestão de direitos e grupos| Os administradores tornam os utilizadores membros de grupos. Os proprietários de aplicações e recursos dão então aos grupos acesso a apps ou recursos.| [Os grupos](https://docs.microsoft.com/azure/active-directory/fundamentals/active-directory-groups-create-azure-portal) também estão disponíveis na AD Azure e os administradores também podem usar grupos para conceder permissões aos recursos. Em Azure AD, os administradores podem atribuir a adesão a grupos manualmente ou usar uma consulta para incluir os utilizadores de forma dinâmica a um grupo. </br> Os administradores podem utilizar a gestão de [direitos](https://docs.microsoft.com/azure/active-directory/governance/entitlement-management-overview) em Azure AD para dar aos utilizadores acesso a uma coleção de apps e recursos usando fluxos de trabalho e, se necessário, critérios baseados no tempo. |
| Gestão de administradores|As organizações utilizarão uma combinação de domínios, unidades organizacionais e grupos em AD para delegar direitos administrativos para gerir o diretório e os recursos que controla.| A Azure AD fornece [funções incorporadas](https://docs.microsoft.com/azure/active-directory/fundamentals/active-directory-users-assign-role-azure-portal) com o seu sistema de controlo de acesso baseado em papéis (RBAC), com suporte limitado para [criar papéis personalizados](https://docs.microsoft.com/azure/active-directory/users-groups-roles/roles-custom-overview) para delegar acesso privilegiado ao sistema de identidade, aplicações e recursos que controla.</br>A gestão de funções pode ser reforçada com [a Privileged Identity Management (PIM)](https://docs.microsoft.com/azure/active-directory/privileged-identity-management/pim-configure) para proporcionar acesso justo no tempo, restrito ao tempo ou ao fluxo de trabalho a papéis privilegiados. |
| Gestão credencial| As credenciais no Diretório Ativo baseiam-se em palavras-passe, autenticação de certificadoe autenticação de smartcard. As palavras-passe são geridas utilizando políticas de palavra-passe baseadas no comprimento da palavra-passe, expiração e complexidade.|A Azure AD utiliza [uma proteção](https://docs.microsoft.com/azure/active-directory/authentication/concept-password-ban-bad) inteligente de senhas para nuvem e no local. A proteção inclui bloqueio inteligente e bloqueio de frases e substituições comuns e personalizadas. </br>A Azure AD aumenta significativamente a segurança através da [autenticação multi-factor](https://docs.microsoft.com/azure/active-directory/authentication/concept-mfa-howitworks) e das tecnologias [sem palavras-passe,](https://docs.microsoft.com/azure/active-directory/authentication/concept-authentication-passwordless) como o FIDO2. </br>A Azure AD reduz os custos de suporte fornecendo aos utilizadores um sistema [de redefinição de senha de autosserviço.](https://docs.microsoft.com/azure/active-directory/authentication/concept-sspr-howitworks) |
| **Aplicações**|||
| Aplicativos de infraestrutura|O Diretório Ativo constitui a base para muitos componentes de infraestruturas no local, por exemplo, DNS, DHCP, IPSec, WiFi, NPS e ACESSO VPN|Num novo mundo na nuvem, o Azure AD, é o novo plano de controlo para aceder a apps versus confiar nos controlos de rede. Quando os utilizadores autenticarem,[o acesso condicional (CA),](https://docs.microsoft.com/azure/active-directory/conditional-access/overview)controlará quais os utilizadores, terá acesso às aplicações em condições exigidas.|
| Aplicativos tradicionais e antigos| A maioria das aplicações no local utilizam LDAP, Autenticação Integrada no Windows (NTLM e Kerberos) ou autenticação baseada em Cabeçalho para controlar o acesso aos utilizadores.| A Azure AD pode fornecer acesso a este tipo de aplicações no local utilizando agentes [proxy de aplicação da Aplicação Azure AD](https://docs.microsoft.com/azure/active-directory/manage-apps/application-proxy) que executam no local. A utilização deste método, a Azure AD pode autenticar utilizadores de Directórioactivo no local utilizando Kerberos enquanto migra ou precisa de coexistir com aplicações antigas. |
| Aplicações SaaS|O Diretório Ativo não suporta as aplicações SaaS de forma nativa e requer sistema de federação, como AD FS.|As aplicações SaaS que suportam a OAuth2, SAML e WS-\* autenticação podem ser integradas para utilizar o Azure AD para autenticação. |
| Aplicativos de linha de negócios (LOB) com autenticação moderna|As organizações podem usar AD FS com Diretório Ativo para apoiar aplicações LOB que requerem autenticação moderna.| As aplicações LOB que requerem autenticação moderna podem ser configuradas para utilizar o Azure AD para autenticação. |
| Serviços de nível médio/Daemon|Os serviços que funcionam em ambientes no local normalmente utilizam contas de serviço AD ou contas de serviço geridas pelo grupo (gMSA) para executar. Estas aplicações herdarão então as permissões da conta de serviço.| A Azure AD fornece [identidades geridas](https://docs.microsoft.com/azure/active-directory/managed-identities-azure-resources/index) para executar outras cargas de trabalho na nuvem. O ciclo de vida destas identidades é gerido pela Azure AD e está ligado ao fornecedor de recursos não pode ser usado para outros fins para obter acesso à porta de trás.|
| **Dispositivos**|||
| Telemóvel|O Diretório Ativo não suporta de forma nativa dispositivos móveis sem soluções de terceiros.| A solução de gestão de dispositivos móveis da Microsoft, microsoft Intune, está integrada com a Azure AD. O Microsoft Intune fornece informações estatais do dispositivo ao sistema de identidade para avaliar durante a autenticação. |
| Desktops windows|O Ative Directy fornece a capacidade de se juntar a dispositivos Windows para os gerir utilizando a Política de Grupo, o System Center Configuration Manager ou outras soluções de terceiros.|Os dispositivos Windows podem ser [unidos ao Azure AD](https://docs.microsoft.com/azure/active-directory/devices/). O acesso condicional pode verificar se um dispositivo é Azure AD aderiu como parte do processo de autenticação. Os dispositivos Windows também podem ser geridos com o [Microsoft Intune](https://docs.microsoft.com/intune/what-is-intune). Neste caso, o acesso condicional, irá considerar se um dispositivo é reclamação (por exemplo, patches de segurança atualizados e assinaturas de vírus) antes de permitir o acesso às aplicações.|
| Servidores Windows| O Ative Directy fornece fortes capacidades de gestão para servidores Windows no local utilizando a Política de Grupo ou outras soluções de gestão.| Os servidores Windows máquinas virtuais em Azure podem ser geridos com serviços de [domínio Azure AD](https://docs.microsoft.com/azure/active-directory-domain-services/). [As identidades geridas](https://docs.microsoft.com/azure/active-directory/managed-identities-azure-resources/index) podem ser utilizadas quando os VMs precisam de acesso ao diretório ou recursos do sistema de identidade.|
| Cargas de trabalho Linux/Unix|O Ative Directory não suporta de forma nativa o não Windows sem soluções de terceiros, embora as máquinas Linux possam ser configuradas para autenticar com o Ative Directory como um reino Kerberos.|Os VMs Linux/Unix podem usar [identidades geridas](https://docs.microsoft.com/azure/active-directory/managed-identities-azure-resources/index) para aceder ao sistema ou recursos de identidade. Algumas organizações migram estas cargas de trabalho para tecnologias de contentores em nuvem, que também podem usar identidades geridas.|
