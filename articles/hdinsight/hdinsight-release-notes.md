# <a name="release-notes-for-azure-hdinsight"></a>Notas de versão do Azure HDInsight

Este artigo fornece informações sobre o **mais recente** atualizações de versões do Azure HDInsight. Para obter informações sobre versões anteriores, consulte [arquivamento de notas de versão do HDInsight](hdinsight-release-notes-archive.md).

> [!IMPORTANT]
> O Linux é o único sistema operativo utilizado na versão 3.4 ou superior do HDInsight. Para obter mais informações, consulte [artigo de controle de versão do HDInsight](hdinsight-component-versioning.md).

## <a name="summary"></a>Resumo

O Azure HDInsight é um dos serviços mais populares entre os clientes empresariais para análise de Hadoop e Spark de código aberto no Azure. Com o [além de cortar no HDInsight de preço de 50 por cento](https://azure.microsoft.com/blog/azure-hdinsight-announcements-significant-price-reduction-and-amazing-new-capabilities/#_blank), os clientes mover para a cloud são colhe poupança ainda maior que nunca.

## <a name="new-features"></a>Novos recursos

As novas atualizações e capacidades se encontra às seguintes categorias:

1.  ***Atualizar o Hadoop e outros projetos de open source*** – além de mais de 1000 correções de bugs em projetos de código-fonte aberto, esta atualização contém uma nova versão de 20 + **Spark (2.3)** e **Kafka (1.0)**.

    a.  [**Novos recursos do Spark 2.3**](https://spark.apache.org/releases/spark-release-2-3-0.html)

    b.  [**Novos recursos do Kafka 1.0**](https://www.apache.org/dist/kafka/1.0.0/RELEASE_NOTES.html)

2.  ***Atualizar o R Server 9.1 para Machine Learning Services 9.3*** – com esta versão, estamos oferecendo os cientistas de dados e engenheiros com o melhor do código-fonte aberto melhorado com inovações algorítmica e a facilidade de operacionalização, todos disponível no respetivo idioma preferencial com a velocidade do Apache Spark. Esta versão expande os recursos oferecidos no R Server com foi adicionado suporte para o Python, que leva para a alteração do nome de cluster do R Server nos serviços de ML. 

3.  ***Suporte para a geração 2 de armazenamento do Azure Data Lake*** – HDInsight irá suportar a versão de pré-visualização de geração 2 de armazenamento do Azure Data Lake. Nas regiões disponíveis, os clientes poderão escolher uma conta do ADLS Gen2 como armazenamento primária ou secundária para os clusters do HDInsight.

4.  ***Atualizações de pacote de segurança do HDInsight Enterprise (pré-visualização)*** – (pré-visualização) [pontos finais de serviço de rede Virtual](https://docs.microsoft.com/en-us/azure/virtual-network/virtual-network-service-endpoints-overview) suporte para armazenamento de Blobs do Azure, ADLS Gen1, Cosmos DB e Azure DB.

### <a name="mahout"></a>Mahout

No HDP 2.3.x e 2.4.x, em vez de envio de uma versão específica do Apache de Mahout, podemos sincronizados para um ponto de revisão específica no Apache Mahout ramal. Este ponto de revisão é depois da 0.9.0 de versão, mas antes do 0.10.0 de versão. Esta opção fornece um grande número de correções e aprimoramentos funcionais sobre a 0.9.0 de versão, mas fornece uma versão estável da funcionalidade Mahout antes da conversão completa para o novo Mahout baseada no Spark no 0.10.0.

O ponto de revisão escolhido para o Mahout no HDP 2.3.x e 2.4.x é do ramo "mahout 0.10.x" do Apache Mahout, a partir de 19 de Dezembro de 2014, revisão 0f037cb03e77c096 no GitHub.

No HDP 2.5.x e 2.6.x, podemos remover a biblioteca de "commons httpclient" de Mahout, porque vamos vê-la como uma biblioteca obsoleta com possíveis problemas de segurança e atualizado o cliente do Hadoop em Mahout para versão 2.7.3, a mesma versão utilizada na HDP 2.5. Como resultado:

-   Anteriormente compiladas Mahout tarefas tem de ser recompilado num ambiente de 2,6 ou HDP-2.5.

-   Há uma pequena possibilidade que algumas tarefas de Mahout poderão encontrar "ClassNotFoundException" ou "não foi possível carregar a classe" erros relacionados com "org.apache.commons.httpclient", "net.java.dev.jets3t" ou prefixos de nome de classe relacionados. Se estes erros, pode considerar se pretende instalar manualmente os jars necessários no caminho da sua classe para a tarefa, se o risco de problemas de segurança na biblioteca do obsoleto é aceitável no seu ambiente.

-   Há uma possibilidade ainda mais pequena do que algumas tarefas de Mahout poderão encontrar falhas em chamadas de código de cliente do hbase do Mahout para as bibliotecas comuns do hadoop, devido a problemas de compatibilidade binária. Infelizmente, não é possível para resolver este problema, exceto pelo fato reverter para a versão de HDP 2.4.2 do Mahout, que pode ter problemas de segurança. Novamente, isso deve ser bastante invulgar e é improvável que ocorram em qualquer determinado conjunto de trabalho de Mahout.

### <a name="cve-2017-7676"></a>**CVE-2017-7676**

| **Resumo:** ignora os carateres após a avaliação de políticas do Apache Ranger '\*"caráter universal |
|----------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Gravidade:** críticas                                                                           |
| **Fornecedor:** Hortonworks                                                                          |
| **Versões Affected:** versões 3.6 do HDInsight, incluindo Apache Ranger versões 0.5.x/0.6.x/0.7.0     |
| **Utilizadores afetados:** ambientes que utilizam políticas do Ranger com caracteres depois '\*"caráter universal – como meu\*testar, testar\*. txt |
| **Impacto:** política recurso na ferramenta de correspondência ignora os carateres após '\*"caráter universal, que pode resultar em comportamentos indesejados.      |
| **Corrigir detalhe:** Ranger política recurso na ferramenta de correspondência foi atualizada para a manipulação correta de correspondências de caráter universal.           |
| **Ação recomendada:** atualizar para HDI 3.6 (com o Apache Ranger 0.7.1+).                                |

    -   [RANGER 1607](https://issues.apache.org/jira/browse/RANGER-1607): solução alternativa para uma falha na verificação do serviço Ranger durante a atualização para o HDP 2.6.2 de versões anteriores do HDP.

        >[!NOTE]
        >Quando Ranger é apenas SSL ativado. 
    
    Este problema surge quando tentar atualizar para o HDP 2.6.1 de versões anteriores do HDP através do Ambari. Ambari utiliza uma chamada de curl para efetuar uma verificação de serviço para serviço de Ranger do Ambari. Se a versão do JDK utilizada pelo Ambari JDK 1.7, a chamada de curl irá falhar com o erro abaixo:
    
    `curl: (35) error:14077410:SSL routines:SSL23_GET_SERVER_HELLO:sslv3 alert handshake failure`
    
    O motivo para este erro é a versão do tomcat utilizada no Ranger é Tomcat 7.0.7\*. Usar o JDK 1.7 está em conflito com cifras predefinido fornecidas no Tomcat 7.0.7\*.
    
    Pode resolver este problema de duas formas:
    
    -   Atualizar o JDK utilizado no Ambari do JDK 1.7 para o JDK 1.8 (consulte a secção [altere a versão do JDK](https://docs.hortonworks.com/HDPDocuments/Ambari-2.2.0.0/bk_ambari_reference_guide/content/ch_changing_the_jdk_version_on_an_existing_cluster.html) no guia de referência do Ambari).
    
    -   Se pretender continuar a suportar um ambiente do JDK 1.7:
    
        1.  Adicionar a propriedade ranger.tomcat.ciphers na secção de site de administração de ranger na sua configuração de Ambari Ranger com o valor inferior:
            
            SSL\_RSA\_WITH\_RC4\_128\_MD5, SSL\_RSA\_WITH\_RC4\_128\_SHA, TLS\_RSA\_ WITH\_AES\_128\_CBC\_SHA, SSL\_RSA\_WITH\_3DES\_EDE\_CBC\_SHA
        
        2.  Se o seu ambiente estiver configurado para o Ranger KMS, adicione a propriedade ranger.tomcat.ciphers na secção de theranger local kms local na sua configuração de Ambari Ranger com o valor inferior:
            
            SSL\_RSA\_WITH\_RC4\_128\_MD5, SSL\_RSA\_WITH\_RC4\_128\_SHA, TLS\_RSA\_ WITH\_AES\_128\_CBC\_SHA, SSL\_RSA\_WITH\_3DES\_EDE\_CBC\_SHA
    
    >[!NOTE]
    >Os valores indicados trabalham exemplos e podem não ser indicativos do seu ambiente. Certifique-se de que corresponde à forma como definir essas propriedades como o seu ambiente estiver configurado.   

-   **RangerUI: Escape de texto de condição de política introduzido no formulário da política**
    
    **O componente afetado:** Ranger
    
    **Descrição do problema**
    
    Se um usuário deseja criar a política com as condições de política personalizada e o texto ou uma expressão contém carateres especiais, imposição de política não irá funcionar. Carateres especiais são convertidos em ASCII antes de guardar a política no banco de dados.
    
    **Carateres especiais:**  &  &lt; &gt; " \` '
    
    Por exemplo, o tags.attributes de condição\['type'\]= "abc" seria convertida para o seguinte depois da política ser guardada.
    
    tags.attds\[&\#x27; dsds &\#x27;\] =& \#x27; cssdfs &\#x27;
    
    Pode ver a condição de política com estes carateres ao abrir a política no modo de edição.
    
    **Solução**
    
    - **Opção \#1: criar/atualizar a política através da API de Rest do Ranger**
        
        URL de REST: http://&lt;anfitrião&gt;: 6080/service/plug-ins/políticas
        
        **Criar política com a condição de política:**

## <a name="deprecation"></a>Preterição

-   **Portal do OMS:** foi removida a ligação da página de recursos do HDInsight que foi apontando para o portal do OMS. O log Analytics utilizado inicialmente o seu próprio portal chamado no portal do OMS para gerir a sua configuração e analisar os dados recolhidos. Toda a funcionalidade deste portal foi movida para o portal do Azure onde irá continuar a ser desenvolvidas. HDInsight preteriu o suporte para o portal do OMS. Os clientes irão utilizar a integração de HDInsight Log Analytics no portal do Azure.

-   **2.3 do Spark**

    -   <https://spark.apache.org/releases/spark-release-2-3-0.html#deprecations>

## <a name="upgrading"></a>A atualizar

Todas estas funcionalidades estão disponíveis no HDInsight 3.6. Para obter a versão mais recente do Spark, Kafka e R Server (Machine Learning Services), escolha a versão do Spark, Kafka, serviços de ML quando [criar um cluster do HDInsight 3.6](https://docs.microsoft.com/en-us/azure/hdinsight/hdinsight-hadoop-provision-linux-clusters). Para obter suporte para ADLS, pode escolher o tipo de armazenamento do ADLS como uma opção. Clusters existentes não serão ser atualizados automaticamente para essas versões.

Todos os clusters nova criados depois de Junho de 2018 obterá automaticamente as correções 1000 + em todos os projetos de código-fonte aberto. Siga [isso](https://docs.microsoft.com/en-us/azure/hdinsight/hdinsight-upgrade-cluster) guia de práticas recomendadas sobre a atualização para uma versão mais recente do HDInsight.
