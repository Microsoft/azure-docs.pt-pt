### <a name="new-features"></a>Novas funcionalidades

#### <a name="service-tags"></a>Etiquetas de serviço
As etiquetas de serviço simplificam a segurança das máquinas virtuais Azure e das redes virtuais Azure, permitindo-lhe restringir facilmente o acesso à rede aos serviços Azure. Pode utilizar etiquetas de serviço nas regras do seu grupo de segurança de rede (NSG) para permitir ou negar o tráfego a um serviço Azure específico a nível global ou por região do Azure. O Azure fornece a manutenção de endereços IP subjacentes a cada etiqueta. As etiquetas de serviço HDInsight para grupos de segurança de rede (NSGs) são grupos de endereços IP para serviços de saúde e gestão. Estes grupos ajudam a minimizar a complexidade da criação de regras de segurança. Os clientes HDInsight podem ativar a etiqueta de serviço através do Portal Azure, PowerShell e REST API. Para mais informações, consulte as etiquetas de serviço do Grupo de [Segurança da Rede (NSG) para o Azure HDInsight](https://docs.microsoft.com/azure/hdinsight/hdinsight-service-tags).

#### <a name="custom-ambari-db"></a>Ambari DB Personalizado
O HDInsight agora permite-lhe usar o seu próprio SQL DB para Apache Ambari. Pode configurar este Ambari DB personalizado a partir do portal Azure ou através do modelo de gestor de recursos.  Esta funcionalidade permite-lhe escolher o SQL DB certo para as suas necessidades de processamento e capacidade. Também pode atualizar facilmente para corresponder aos requisitos de crescimento do negócio. Para mais informações, consulte [Configurar os clusters HDInsight com um Ambari DB personalizado](hdinsight-custom-ambari-db.md).

![Ambari DB Personalizado](./media/hdinsight-release-notes/custom-ambari-db.png)

#### <a name="transport-layer-security-tls-12-enforcement"></a>Segurança da Camada de Transporte (TLS) 1.2 execução
A Segurança da Camada de Transporte (TLS) e a Camada de Tomadas Seguras (SSL) são protocolos criptográficos que fornecem segurança de comunicações sobre uma rede informática. Para mais informações, consulte [Transport Layer Security](https://en.wikipedia.org/wiki/Transport_Layer_Security#SSL_1.0.2C_2.0_and_3.0). Enquanto os clusters Azure HDInsight aceitam ligações TLS 1.2 em pontos finais HTTPS públicos, o TLS 1.1 ainda é suportado para retrocompatibilidade com clientes mais velhos.

A partir do próximo lançamento, poderá optar por optar e configurar os seus novos clusters HDInsight para apenas aceitar ligações TLS 1.2. 

No final do ano, a partir de 6/30/2020, o Azure HDInsight aplicará versões TLS 1.2 ou posteriores para todas as ligações HTTPS. Recomendamos que garanta que todos os seus clientes estão prontos para lidar com versões TLS 1.2 ou posteriores.

#### <a name="moving-to-azure-virtual-machine-scale-sets"></a>Movendo-se para conjuntos de escala de máquinas virtuais Azure

Veja mais sobre os conjuntos de escala de [máquinas virtuais Azure](https://docs.microsoft.com/azure/virtual-machine-scale-sets/overview).

### <a name="new-features"></a>Novas funcionalidades

#### <a name="hdinsight-identity-broker-hib-preview"></a>HDInsight Identity Broker (HIB) (Pré-visualização)

O HDInsight Identity Broker (HIB) permite que os utilizadores instiem a Apache Ambari usando a autenticação multi-factor (MFA) e obtenha os bilhetes Kerberos necessários sem precisar de hashes de senha nos Serviços de Domínio de Diretório Ativo azure (AAD-DS). Atualmente, o HIB só está disponível para clusters implantados através do modelo azure de gestão de recursos (ARM).

#### <a name="kafka-rest-api-proxy-preview"></a>Kafka Rest API Proxy (Pré-visualização)

Kafka Rest API Proxy fornece uma implantação de um clique de proxy REST altamente disponível com cluster Kafka através de autorização AAD segura e protocolo OAuth. 

#### <a name="auto-scale"></a>Dimensionar automaticamente

A escala automática para Azure HDInsight está agora geralmente disponível em todas as regiões para os tipos de cluster Apache Spark e Hadoop. Esta funcionalidade permite gerir as cargas de trabalho de análise de big data de uma forma mais rentável e produtiva. Agora pode otimizar o uso dos seus clusters HDInsight e apenas pagar pelo que precisa.

Dependendo dos seus requisitos, pode escolher entre a autoscalcificação baseada em carga e a programação. A Escala Automática baseada em carga pode escalar o tamanho do cluster para cima e para baixo com base nas necessidades atuais de recursos, enquanto a escala automática baseada no horário pode alterar o tamanho do cluster com base num horário predefinido. 

Suporte em escala automática para a carga de trabalho HBase e LLAP também é pré-visualização pública. Para mais informações, consulte automaticamente a escala dos [clusters Azure HDInsight](https://docs.microsoft.com/azure/hdinsight/hdinsight-autoscale-clusters).

#### <a name="hdinsight-accelerated-writes-for-apache-hbase"></a>HDInsight Accelerated Writes para Apache HBase 

As Escritas Aceleradas utilizam os discos geridos Premium SSD do Azure para melhorar o desempenho do Registo de Escrita Antecipada (WAL) do Apache HBase. Para mais informações, veja [Escritas Aceleradas do Azure HDInsight para o Apache HBase](https://docs.microsoft.com/azure/hdinsight/hbase/apache-hbase-accelerated-writes).

#### <a name="custom-ambari-db"></a>Ambari DB Personalizado

A HDInsight oferece agora uma nova capacidade para permitir que os clientes utilizem o seu próprio SQL DB para ambari. Agora, os clientes podem escolher o SQL DB certo para ambari e facilmente atualizá-lo com base no seu próprio requisito de crescimento do negócio. A implementação é feita com um modelo de Gestor de Recursos Azure. Para mais informações, consulte [Configurar os clusters HDInsight com um Ambari DB personalizado](https://docs.microsoft.com/azure/hdinsight/hdinsight-custom-ambari-db).

#### <a name="f-series-virtual-machines-are-now-available-with-hdinsight"></a>Máquinas virtuais da série F já estão disponíveis com o HDInsight

As máquinas virtuais da série F (VMs) são uma boa escolha para começar com o HDInsight com requisitos de processamento de luz. A um preço de tabela por hora mais baixo, a série F é o melhor valor de desempenho por preço no portefólio do Azure com base na Unidade de Computação do Azure (ACU) por vCPU. Para mais informações, consulte [A Seleção do tamanho de VM certo para o seu cluster Azure HDInsight](https://docs.microsoft.com/azure/hdinsight/hdinsight-selecting-vm-size).

### <a name="known-issues"></a>Problemas conhecidos

#### <a name="replication-is-broken-for-secure-hbase-with-default-installation"></a>A replicação está partida para Secure HBase com instalação predefinida

Para o HDInsight 4.0, faça os seguintes passos:

1. Ativar a comunicação inter-cluster.
1. Inscreva-se no nódeio ativo.
1. Descarregue um script para permitir a replicação com o seguinte comando:

    ```
    sudo wget https://raw.githubusercontent.com/Azure/hbase-utils/master/replication/hdi_enable_replication.sh
    ```
1. Escreva o comando `sudo kinit <domainuser>`.
1. Digite o seguinte comando para executar o script:

    ```
    sudo bash hdi_enable_replication.sh -m <hn0> -s <srclusterdns> -d <dstclusterdns> -sp <srcclusterpasswd> -dp <dstclusterpasswd> -copydata
    ```
Para o HDInsight 3.6, faça o seguinte:

1. Inscreva-se na HMaster ZK ativa.
1. Descarregue um script para permitir a replicação com o seguinte comando:
    ```
    sudo wget https://raw.githubusercontent.com/Azure/hbase-utils/master/replication/hdi_enable_replication.sh
    ```
1. Escreva o comando `sudo kinit -k -t /etc/security/keytabs/hbase.service.keytab hbase/<FQDN>@<DOMAIN>`.
1. Escreva o seguinte comando:

    ```bash
    sudo bash hdi_enable_replication.sh -s <srclusterdns> -d <dstclusterdns> -sp <srcclusterpasswd> -dp <dstclusterpasswd> -copydata
    ```

#### <a name="phoenix-sqlline-stops-working-after-migrating-hbase-cluster-to-hdinsight-40"></a>Phoenix Sqlline deixa de funcionar após migrar o cluster HBase para HDInsight 4.0

Faça os seguintes passos:

1. Largue as seguintes tabelas Phoenix:
    1. `SYSTEM.FUNCTION`
    1. `SYSTEM.SEQUENCE`
    1. `SYSTEM.STATS`
    1. `SYSTEM.MUTEX`
    1. `SYSTEM.CATALOG`
1. Se não conseguir eliminar nenhuma das tabelas, reinicie a HBase para limpar quaisquer ligações às tabelas.
1. Execute novamente `sqlline.py`. Phoenix vai recriar todas as tabelas que foram apagadas no passo 1.
1. Regenerar as tabelas e vistas da Phoenix para os seus dados hbase.

#### <a name="phoenix-sqlline-stops-working-after-replicating-hbase-phoenix-metadata-from-hdinsight-36-to-40"></a>Phoenix Sqlline deixa de funcionar após replicar metadados HBase Phoenix de HDInsight 3.6 a 4.0

#### <a name="mahout"></a>Mahout

Em HDP-2.3.x e 2.4.x, em vez de enviar uma versão apache específica de Mahout, sincronizamos para um ponto de revisão particular no tronco apache Mahout. Este ponto de revisão é depois do lançamento de 0.9.0, mas antes do lançamento de 0.10.0. Isto fornece um grande número de correções de bugs e melhorias funcionais durante o lançamento 0.9.0, mas fornece uma libertação estável da funcionalidade Mahout antes da conversão completa para o novo Mahout baseado em Spark em 0.10.0.

O ponto de revisão escolhido para Mahout em HDP 2.3.x e 2.4.x é do ramo "mahout-0.10.x" de Apache Mahout, a partir de 19 de dezembro de 2014, revisão 0f037cb03e7c096 no GitHub.

Em HDP-2.5.x e 2.6.x, removemos a biblioteca "commons-httpclient" de Mahout porque a vemos como uma biblioteca obsoleta com possíveis problemas de segurança, e atualizámos o Hadoop-Client em Mahout para a versão 2.7.3, a mesma versão usada no HDP-2.5. Como resultado:

-   Os postos de trabalho de Mahout previamente compilados terão de ser recompilados no ambiente HDP-2,5 ou 2,6.

-   Existe uma pequena possibilidade de alguns empregos de Mahout poderem encontrar erros "ClassNotFoundException" ou "não podiam carregar classe" relacionados com "org.apache.commons.httpclient", "net.java.dev.jets3t", ou prefixos de nome de classe relacionados. Se estes erros ocorrerem, poderá considerar se instala manualmente os frascos necessários no seu caminho de classe para o trabalho, se o risco de problemas de segurança na biblioteca obsoleta for aceitável no seu ambiente.

-   Existe uma possibilidade ainda menor de que alguns empregos de Mahout possam encontrar falhas nas chamadas de código de cliente hbase de Mahout para as bibliotecas comuns de hadoop, devido a problemas de compatibilidade binários. Infelizmente, não há forma de resolver esta questão, a não ser reverter para a versão HDP-2.4.2 de Mahout, que pode ter problemas de segurança. Mais uma vez, isto deve ser muito incomum, e é improvável que ocorra em qualquer suíte de trabalho mahout.

      > [!NOTE]  
      > Só quando o Ranger estiver ativado pela SSL. 
     
  Esta questão surge quando se tenta atualizar para HDP-2.6.1 a partir de versões anteriores do HDP através de Ambari. Ambari usa uma chamada de caracóis para fazer uma verificação de serviço ao serviço Ranger em Ambari. Se a versão JDK utilizada por Ambari for JDK-1.7, a chamada de caracóis falhará com o erro abaixo:
    
  `curl: (35) error:14077410:SSL routines:SSL23_GET_SERVER_HELLO:sslv3 alert handshake failure`
    
  A razão para este erro é que a versão tomcat utilizada\*no Ranger é Tomcat-7.0.7 . A utilização de conflitos JDK-1.7 com cifras padrão\*fornecidas em Tomcat-7.0.7 .
    
  Pode resolver esta questão de duas formas:
    
  -   Atualize o JDK utilizado em Ambari de JDK-1.7 para JDK-1.8 (consulte a secção [Alterar a versão JDK](https://docs.hortonworks.com/HDPDocuments/Ambari-2.2.0.0/bk_ambari_reference_guide/content/ch_changing_the_jdk_version_on_an_existing_cluster.html) no Guia de Referência Ambari).
    
  -   Se quiser continuar a apoiar um ambiente JDK-1.7:
    
      1.  Adicione o guarda-propriedades.tomcat.ciphers na secção ranger-admin-site na sua configuração Ambari Ranger com o valor abaixo:
            
          SSL\_\_RSA\_COM\_RC4\_128 MD5, SSL\_RSA\_COM\_\_RC4 128\_SHA, TLS\_Rsa\_COM\_\_\_\_\_\_AES\_128\_CBC\_SHA, SSL\_RSA COM 3DES EDE CBC SHA
        
      2.  Se o seu ambiente estiver configurado para Ranger-KMS, adicione os property ranger.tomcat.ciphers na secção theranger-kms-site na sua configuração Ambari Ranger com o valor abaixo:
            
          SSL\_\_RSA\_COM\_RC4\_128 MD5, SSL\_RSA\_COM\_\_RC4 128\_SHA, TLS\_Rsa\_COM\_\_\_\_\_\_AES\_128\_CBC\_SHA, SSL\_RSA COM 3DES EDE CBC SHA
    
  >[!NOTE]  
  >Os valores notados são exemplos de trabalho e podem não ser indicativos do seu ambiente. Certifique-se de que a forma como define estas propriedades corresponde à configuração do seu ambiente.   

- **RangerUI: Texto de fuga à condição política inscrito na forma política**
    
  **Componente afetado:** Ranger
    
  **Descrição do Problema**
    
  Se um utilizador quiser criar uma política com condições políticas personalizadas e a expressão ou texto contiver caracteres especiais, então a aplicação da política não funcionará. Os caracteres especiais são convertidos em ASCII antes de guardar a apólice na base de dados.
    
  **Personagens especiais:**  &  &lt; &gt; " \`
    
  Por exemplo, as etiquetas\[de\]condição.atributos 'type' ='abc' seriam convertidas para o seguinte uma vez que a apólice é guardada.
    
  tags.attds\[&\#x27;dsds \#&x27; \] \#x27;cssdfs&x27; =& \#

### <a name="deprecation"></a>Preterição

-   **Portal OMS:** Removemos o link da página de recursos HDInsight que apontava para o portal OMS. ### <a name="upgrading"></a>Upgrade

Todas estas funcionalidades estão disponíveis no HDInsight 3.6. Por favor, siga [este](https://docs.microsoft.com/azure/hdinsight/hdinsight-upgrade-cluster) guia para as melhores práticas em torno da atualização para uma versão HDInsight mais recente.
